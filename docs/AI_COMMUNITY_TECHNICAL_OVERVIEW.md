# Slack File Previewer 技術解説

AI活用コミュニティ共有用の技術解説です。エンジニアでない人にも伝わるように、「何を解決するのか」「裏側で何が起きているのか」「AIを使った開発で何がよかったのか」を中心にまとめます。

## ひとことで言うと

SlackにMarkdownやHTMLファイルをアップロードすると、botが自動で中身を読み取り、スレッドに見やすいプレビューリンクを投稿するアプリです。

たとえば、チーム内で以下のようなファイルを共有するとします。

- 議事録
- 仕様書
- プロンプト集
- AIエージェントへの作業指示書
- WebページのHTML断片
- Markdownで書いたドキュメント

通常は、Slack上では中身を確認しづらく、いちいちダウンロードして開く必要があります。このアプリは、その手間を減らします。

## ユーザー体験

利用者がやることはシンプルです。

1. Slackチャンネルに `.md` や `.html` ファイルをアップロードする
2. botがスレッドにプレビューを投稿する
3. プレビューリンクを開いて内容を見る

Markdownプレビューでは、以下のような読みやすさのための機能があります。

- 見出しから自動生成される目次
- テーブルのスタイリング
- 数字リストの整形
- 区切り線やリンクの見やすい表示
- Markdown全文をコピーするボタン
- Google Docsを開いて貼り付けやすくするボタン

HTMLファイルは安全性を優先し、そのままWebページとして実行せず、テキストとして安全に表示します。

## なぜ作ったのか

AIを使ったチーム作業では、Markdownファイルがとても増えます。

- 仕様書
- プロンプト
- エージェントへの指示
- 作業ログ
- 検証結果
- レビューコメント

しかしSlackにMarkdownを置くだけでは、本文の構造やテーブルが見えづらくなります。特にAIエージェントと共同作業していると、ファイルの中身をすばやく確認できることが作業速度に直結します。

このアプリは、「Slackで共有されたテキスト資産を、その場で読む」ための小さなインフラです。

## 全体の仕組み

大きな流れは次の通りです。

```text
Slackにファイルが投稿される
  ↓
SlackがWorkerへイベントを送る
  ↓
Workerがイベントを検証する
  ↓
WorkflowがSlackからファイルを取得する
  ↓
Markdown / HTMLを安全なプレビューHTMLに変換する
  ↓
R2にプレビューHTMLを保存する
  ↓
D1にメタデータを保存する
  ↓
SlackスレッドへプレビューURLを投稿する
```

登場する技術は多く見えますが、役割で見ると単純です。

| 役割 | 使っているもの | 何をしているか |
|---|---|---|
| Slackとの入口 | Slack Events API | ファイル共有や削除を検知する |
| サーバー処理 | Cloudflare Workers | Slackから来たリクエストを受ける |
| 非同期処理 | Cloudflare Workflows | ファイル取得、変換、保存、投稿を順番に実行する |
| データベース | Cloudflare D1 | プレビューの状態やインストール情報を保存する |
| ファイル保存 | Cloudflare R2 | 生成したHTMLプレビューを保存する |
| 認証 | Slack OAuth | 複数ワークスペースにインストールできるようにする |

## なぜCloudflare Workersなのか

このアプリは、常に大きなサーバーを動かしておく必要がありません。

Slackからイベントが来たときだけ動けばよいので、Cloudflare Workersのようなサーバーレス環境と相性がよいです。

Cloudflareを選ぶメリットは次の通りです。

- 低コストで動かしやすい
- SlackからのWebhookをすばやく受けられる
- D1、R2、Workflowsを同じCloudflare上でまとめられる
- プレビューURLをWorker経由で安全に配信できる

## なぜWorkflowsを使うのか

Slackのイベント受信処理では、すぐにSlackへ応答する必要があります。

しかし、実際のプレビュー作成には複数の処理があります。

- Slack APIでファイル情報を取得する
- private fileをダウンロードする
- MarkdownやHTMLを変換する
- R2へ保存する
- D1へ記録する
- Slackへ返信する

これをWebhookの中で全部やると、処理が遅くなったり、Slack側のリトライで二重投稿が起きたりしやすくなります。

そこで、入口のWorkerは「受け付けたらすぐ応答」し、重い処理はWorkflowへ渡します。これは、人間の仕事でいうと「受付」と「作業担当」を分けるような設計です。

## 安全性の考え方

このプロジェクトでは、単に便利にするだけでなく、Slack上のファイルを不用意に公開しないことを重視しています。

主な安全設計は次の通りです。

| 安全設計 | 内容 |
|---|---|
| Slack署名検証 | 本当にSlackから来たリクエストか確認する |
| private URLを隠す | SlackのファイルURLをブラウザやSlackメッセージへ出さない |
| R2はprivate | 生成したHTMLは直接公開せず、Worker経由で配信する |
| 署名付きURL | プレビューURLに期限付きtokenを付ける |
| 期限管理 | 現在は7日間でプレビューURLが期限切れになる |
| 削除連動 | Slackで削除や共有解除されたファイルはプレビューを無効化する |
| HTMLは実行しない | HTMLファイルは生HTMLとして配信せず、テキスト化して表示する |
| bot token暗号化 | 複数workspace用のbot tokenはD1に暗号化して保存する |

特にHTMLは危険です。HTMLをそのまま表示すると、スクリプト実行や外部通信などのリスクがあります。そのため、現在の実装では「安全なテキストプレビュー」に寄せています。

## 複数ワークスペース対応

最初の実装は、1つのSlackワークスペースだけを想定していました。その場合、Cloudflare Secretにbot tokenを1つ保存すれば動きます。

しかし、2つ以上のワークスペースで継続利用するなら、この方式は破綻します。ワークスペースごとにbot tokenが違うためです。

現在はSlack OAuthに対応しています。

```text
/slack/install を開く
  ↓
Slackの許可画面でワークスペースを選ぶ
  ↓
SlackからOAuth callbackが返る
  ↓
bot tokenを取得する
  ↓
D1にworkspace別installationとして保存する
  ↓
以後、そのworkspaceのイベントでは対応するtokenを使う
```

これにより、同じWorkerを使いながら、複数のSlackワークスペースで動かせます。

## データの持ち方

このアプリは、すべてをデータベースに詰め込むのではなく、用途ごとに保存先を分けています。

| データ | 保存先 | 理由 |
|---|---|---|
| SlackイベントID | D1 | 同じイベントを二重処理しないため |
| プレビューのメタデータ | D1 | 有効期限、状態、ファイルIDを管理するため |
| 生成済みHTML | R2 | 本文HTMLは大きくなるため、オブジェクトストレージに置く |
| workspace別bot token | D1 | ワークスペースごとにSlack APIを呼ぶため |
| 元ファイル本文 | 原則保存しない | 情報漏えい面積を小さくするため |

「必要なものだけ保存し、不要なものは持たない」という方針です。

## AIエージェント開発でよかったこと

このプロジェクトは、AIエージェントと対話しながら段階的に作っています。

特によかったのは、以下のような進め方です。

### 1. 小さく動くMVPから始める

最初から完璧なSlackアプリを作るのではなく、まずはMarkdownファイルを安全にプレビューするところから始めました。

その後、ユーザーが実際に触って気づいた点を追加しました。

- テーブルを見やすくしたい
- 数字リストが改行されない
- 区切り線を罫線として出したい
- 目次がほしい
- Markdownをコピーしたい
- Google Docsで開きたい
- プレビュー期限を24時間から7日にしたい
- 複数ワークスペースで使いたい

AIエージェントには、このような小さな改善を連続で依頼しやすいです。

### 2. テストを書きながら進める

新しい機能を入れるたびに、Vitestでテストを追加しています。

たとえば以下をテストしています。

- Markdownが安全にHTMLへ変換される
- テーブルやリストが正しく表示される
- 危険なリンクをそのまま出さない
- Slackの削除イベントでプレビューを無効化する
- OAuth callbackでworkspace installationを保存する
- bot tokenが暗号化され、復号できる
- workspace別tokenが優先され、legacy tokenはfallbackになる

AIに実装を任せる場合でも、テストがあると「動いたつもり」を減らせます。

### 3. 実サービスの境界をAIと一緒に詰める

このプロジェクトでは、単なる画面作成だけでなく、実サービス特有の境界が多くあります。

- Slack署名検証
- OAuth redirect URL
- workspaceごとのtoken
- private channelでのbot invite
- 期限付きURL
- 削除やunshareとの連動
- Cloudflare secrets
- D1 migration
- R2 private bucket

AIエージェントは、こうした境界を一つずつコード、設定、ドキュメント、テストに落とし込む作業に向いていました。

## AI活用プロジェクトとしての学び

このプロジェクトから得られた学びは、次の3つです。

### 小さい業務摩擦ほど、AIで素早くプロダクト化しやすい

「Slackに上がったMarkdownが読みづらい」という課題は小さいですが、毎日発生すると大きな摩擦になります。

AIエージェントを使うと、こうした小さな業務課題をその場でプロダクト化しやすくなります。

### AIに任せるほど、検証の仕組みが重要になる

AIは実装を速く進められますが、外部サービス連携では思い込みも起きます。

そのため、以下のような検証が重要でした。

- テストを通す
- Cloudflareへ実際にデプロイする
- Slackで実際にファイルを共有する
- エラー画面を見て設定を直す
- GitHub Actionsで継続的に確認する

### ドキュメントも実装の一部になる

Slack App、Cloudflare secrets、OAuth、ワークスペース配布は、コードだけでは完結しません。

人間が設定する部分が多いため、ドキュメントを整えることも実装の一部です。

このリポジトリでは、使い方、API、セキュリティ、アーキテクチャ、エージェント向けhandoffを分けて残しています。

## 今後の改善案

現時点でも実用できますが、さらに進めるなら以下が候補です。

| 改善案 | 目的 |
|---|---|
| Cloudflare Access / Slack OIDC | プレビュー閲覧者をSlackログインで制限する |
| HTMLスクリーンショット化 | HTMLを安全に見た目として確認する |
| Slack token rotation対応 | 長期運用時のtoken管理を強化する |
| App Unfurl対応 | Slack上でリンク展開のように見せる |
| Mermaid / PlantUML対応 | 技術ドキュメントの図もプレビューできるようにする |
| 管理画面 | インストール済みworkspaceやログを見やすくする |

## まとめ

Slack File Previewerは、派手なAIアプリではありません。

しかし、AI時代に増えるMarkdownや仕様書、プロンプト、作業ログを、チームの会話の中で読みやすくするための実用的なツールです。

ポイントは、AIを「チャットで答えを出す存在」としてだけでなく、「日々の小さな業務摩擦を実装へ落とす共同開発者」として使っていることです。

小さな不便を見つけ、すぐに実装し、実環境で検証し、改善を積み重ねる。そういうAI活用の具体例として、このプロジェクトは共有しやすい題材です。
